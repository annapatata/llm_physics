# How to Compute P(next token = t | prefix) Using the DP Table

## The Core Problem

The inside algorithm gives you:

```
alpha[i, j, A] = P(A generates exactly x[i..j])
```

For a **complete** string of known length n, `alpha[0, n-1, root]` is the string probability. Done.

But you want P(next = t | prefix x[0..c-1]). You don't know what comes after position c. The string isn't finished. You can't run inside on an incomplete string and get the answer directly.

---

## Why Inside Alone Is Not Enough

Suppose your prefix is `[1, 3, 2]` (length 3). You want P(next = 1 | [1, 3, 2]).

The definition of this probability is:

```
P(next = 1 | [1,3,2]) = Σ_{all complete strings s ∈ L(G) that start with [1,3,2,1]} P(s)
                         ─────────────────────────────────────────────────────────────
                         Σ_{all complete strings s ∈ L(G) that start with [1,3,2]}   P(s)
```

Both numerator and denominator are **sums over infinitely many complete strings** — all the valid continuations. You can't enumerate them. You need a smarter way.

The standard inside algorithm computes P(A =>* some specific complete string). What you need is P(A =>* any string that starts with some prefix). These are different things.

---

## The Key Insight: Splitting the Derivation

For a binary rule `A → B C` (after binarization), A's derivation splits the string into two parts:

```
x[0 .. k]  generated by B
x[k+1 .. ?]  generated by C
```

For a **known prefix** y = x[0..c] (length c+1, where x[c] = t is our candidate next token):

- If k < c: B generates a **complete, known** subspan y[0..k]. We can use the alpha table for this: `alpha[0, k, B]`. Then C needs to generate y[k+1..c] as a prefix of whatever it ultimately produces.
- If k = c: B generates the entire known prefix y[0..c] completely. Then C generates the **unknown continuation** — anything. Since all our NTs are productive, P(C generates something) = 1.0.

This split gives us a clean recursion.

---

## The Algorithm

### Step 1: Run inside on the extended prefix

For each candidate terminal t ∈ {1, 2, 3}:

Form `y = x[0..c-1] + [t]`, a string of length `m = c+1`.

Run the inside algorithm on y:

```
alpha_y[i, j, A] = P(A generates exactly y[i..j])
```

This is exactly the existing `string_prob` code, just applied to the extended prefix `y` instead of a complete string. You can reuse `alpha[i,j,A]` from the original prefix for all spans that don't touch position c, and only compute the new spans involving position c.

### Step 2: Define the `right` function

```
right(A, start) = P(A generates some string that starts with y[start .. m-1])
```

This asks: "starting at position `start`, can nonterminal A produce the remaining known prefix (y[start] through y[m-1] = t), and then generate anything after that?"

**Base cases:**

```
right(A, start) = 1.0     if start >= m
                           # Nothing left to match. A generates any continuation. = 1.0 because all NTs are productive.

right(PT_t', start) = 1.0  if start == m-1 and y[m-1] == t'
                  = 0.0   otherwise
                           # PT_t' is a preterminal that generates exactly terminal t'.
                           # It can only "start with y[start..m-1]" if the remaining prefix
                           # is exactly [t'] — a single token matching t'.
```

**Recursive case (binary rule A → B C, prob p):**

```
right(A, start) = Σ_{rules A→BC} p × Σ_{k=start}^{m-1} alpha_y[start, k, B] × right(C, k+1)
```

Read this as: B generates y[start..k] **exactly** (captured by alpha_y), and then C takes over from position k+1 and handles the rest of the known prefix. When k = m-1, right(C, m) = 1.0, meaning C just needs to generate any valid continuation.

### Step 3: The unnormalized probability for each t

```
unnorm(t) = right(root, 0)
          = Σ_{root→BC} p × Σ_{k=0}^{m-1} alpha_y[0, k, B] × right(C, k+1)
```

This is the total probability mass of all complete strings in L(G) that start with y = x[0..c-1]+t.

### Step 4: Normalize

```
P(next = t | x[0..c-1]) = unnorm(t) / Σ_{t' ∈ {1,2,3}} unnorm(t')
```

(If EOS is a valid next token in your setup, include it too.)

## Worked Example (Tiny Grammar)

Say the grammar has rules:
```
root → A B   (prob 1.0)
A    → [1]   (prob 0.5),  A → [1, 2] (prob 0.5)
B    → [3]   (prob 1.0)
```

Strings in L(G): `[1, 3]` (prob 0.5) and `[1, 2, 3]` (prob 0.5).

Prefix: `[1]` (c=1). What is P(next = 2 | [1])?

**For t=2, extended prefix y = [1, 2], m=2:**

Inside on y:
```
alpha_y[0, 0, PT_1] = 1.0   (PT_1 generates token 1)
alpha_y[1, 1, PT_2] = 1.0   (PT_2 generates token 2)
alpha_y[0, 1, A]    = 0.5   (A → [1,2] with prob 0.5)
alpha_y[0, 0, A]    = 0.5   (A → [1] with prob 0.5)
alpha_y[1, 1, B]    = 0.0   (B generates [3], not [2])
```

right(root, 0) with y=[1,2]:
```
= 1.0 × [
    Σ_{k=0,1} alpha_y[0, k, A] × right(B, k+1)
  ]
= alpha_y[0, 0, A] × right(B, 1) + alpha_y[0, 1, A] × right(B, 2)
= 0.5 × right(B, 1)               + 0.5 × 1.0

right(B, 1): B generates [3]. y[1] = 2 ≠ 3, and k=m-1=1. → right(B, 1) = 0.0

= 0.5 × 0.0 + 0.5 × 1.0 = 0.5
```

**For t=3, extended prefix y = [1, 3], m=2:**

alpha_y[0,1,A] = 0.0 (A can't generate [1,3] — its rules are [1] and [1,2])
alpha_y[0,0,A] = 0.5

right(root, 0) with y=[1,3]:
```
= alpha_y[0, 0, A] × right(B, 1) + alpha_y[0, 1, A] × right(B, 2)
= 0.5 × right(B, 1) + 0.0 × 1.0

right(B, 1): B generates [3]. y[1] = 3 = 3, k=m-1. → right(B, 1) = 1.0

= 0.5 × 1.0 = 0.5
```

**Normalize:**
```
P(next = 2 | [1]) = 0.5 / (0.5 + 0.5) = 0.5
P(next = 3 | [1]) = 0.5 / (0.5 + 0.5) = 0.5
```

Correct! The string [1,3] has prob 0.5 and [1,2,3] has prob 0.5. Given prefix [1], the next token is 2 (continuing to [1,2,3]) or 3 (completing [1,3]), each with probability 0.5.

## Implementation

```python
def next_token_distribution(prefix: list, cfg, terminals=(1, 2, 3)) -> dict:
    """
    Returns {t: P(next=t | prefix)} for each candidate terminal t.
    
    Uses:
      - inside algorithm on each extended prefix (prefix + [t])
      - right() function for prefix probability
    """
    from dp.binarize import binarize
    from dp.inside import _build_alpha   # you'd refactor string_prob to expose alpha

    bcfg = binarize(cfg)
    unnorm = {}

    for t in terminals:
        y = prefix + [t]
        m = len(y)
        
        # Run inside on extended prefix y
        alpha_y = _build_alpha(y, bcfg)  # shape: (m, m, N)

        # Compute right(A, start) bottom-up on (A, start) pairs
        # right_table[start, A_idx] = right(A, start)
        N = len(nt_list)
        right_table = np.ones((m + 1, N))  # right_table[m, :] = 1.0 by default

        for start in range(m - 1, -1, -1):   # fill from right to left
            for A_id, A in enumerate(nt_list):
                if A in preterminals_for:
                    # A is PT_t', generates exactly terminal preterminals_for[A]
                    t_A = preterminals_for[A]
                    right_table[start, A_id] = 1.0 if (start == m - 1 and y[start] == t_A) else 0.0
                else:
                    # A → B C rules
                    total = 0.0
                    for B, C, p in rules_of[A]:
                        for k in range(start, m):
                            total += p * alpha_y[start, k, B_id] * right_table[k + 1, C_id]
                    right_table[start, A_id] = total

        unnorm[t] = right_table[0, root_id]

    Z = sum(unnorm.values())
    return {t: v / Z for t, v in unnorm.items()} if Z > 0 else {t: 1/len(terminals) for t in terminals}
```

A full clean implementation would refactor `inside.py` to expose the alpha table directly. The structure above shows the logic; wire it up to the binarized CFG the same way `string_prob` does.

## How This Connects to the Paper

**Result 3 (KL divergence):** For each test string x[0..n-1] and each position c, compute:
```
KL(P_true(· | x[0..c-1]) || P_gpt(· | x[0..c-1]))
= Σ_t P_true(t) × log(P_true(t) / P_gpt(t))
```

- `P_gpt(t | prefix)` comes from softmax of the model's logits at position c.
- `P_true(t | prefix)` comes from this algorithm — run for each candidate t.

**Result 1 (generation accuracy):** This algorithm is NOT needed there. You only need CYK to check if the completed string is valid.

## Summary of What Each Function Does

```
alpha[i, j, A]    = P(A generates EXACTLY x[i..j])
                    filled by inside.py, needs the full span to be known

right(A, start)   = P(A generates ANY string starting with y[start..m-1])
                    requires alpha_y for spans within the known prefix
                    uses right_table[k+1, C] for the unknown part beyond position m-1
                    right(A, m) = 1.0  ← "nothing left to constrain, any string works"

unnorm(t)         = right(root, 0) with y = prefix + [t]
                    = total probability mass of all complete strings starting with prefix+t

P(next = t)       = unnorm(t) / sum over t of unnorm(t)
```
